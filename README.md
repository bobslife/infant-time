# infant-time

## 프로젝트 개요

`infant-time`은 신생아의 수유 시간과 배변 시간(대변)을 간편하게 기록하고, 누적 데이터를 바탕으로 요약과 분석 정보를 제공하는 아이폰 중심의 단일 페이지 서비스입니다.

부부가 함께 사용할 수 있도록 복잡한 진입 과정을 줄이고, 가능한 한 앱처럼 빠르고 자연스럽게 사용하는 경험을 목표로 합니다.

## 문제 정의

신생아 돌봄 과정에서는 수유와 배변 기록이 매우 자주 발생하며, 보호자 2인 이상이 번갈아 돌보는 경우가 많습니다. 이때 다음과 같은 문제가 발생합니다.

- 마지막 수유 시점이나 배변 시점을 바로 확인하기 어렵다.
- 배우자와 기록 상태를 공유하기 번거롭다.
- 단순 기록만으로는 패턴을 파악하기 어렵다.
- 웹 링크 기반 서비스는 홈 화면 진입성과 즉시성이 낮아 실제 사용 빈도가 떨어질 수 있다.

따라서 본 프로젝트는 빠른 입력, 즉시 확인 가능한 요약, 아이폰에서 앱처럼 접근 가능한 사용성을 우선 해결해야 합니다.

## 목표

- 아이폰에서 무리 없이 사용할 수 있는 모바일 우선 단일 페이지 서비스를 제공한다.
- 신생아의 수유 시간과 대변 시간을 최소한의 동작으로 기록할 수 있게 한다.
- 기록 데이터를 기반으로 최근 상태 요약과 기초 분석 정보를 제공한다.
- 초기에는 구현 복잡도를 낮추기 위해 내부 저장소 기반으로 시작하되, 향후 외부 저장소로 확장 가능한 구조를 고려한다.
- 홈 화면 추가 등을 통해 앱에 가까운 사용 경험을 제공한다.
- 마지막 수유 후 4시간 동안 기록 갱신이 없을 경우 알림을 제공하는 방향을 검토한다.

## 핵심 기능

- 수유 시간 기록
  - 현재 시각 기준으로 빠르게 입력
  - 필요 시 수동 시간 보정 가능
- 대변 시간 기록
  - 현재 시각 기준으로 빠르게 입력
  - 필요 시 수동 시간 보정 가능
- 최근 기록 조회
  - 마지막 수유 시간
  - 마지막 대변 시간
  - 오늘의 수유 횟수
  - 오늘의 대변 횟수
- 요약 및 분석
  - 최근 수유 간격
  - 일별 수유 횟수
  - 최근 배변 패턴 요약
  - 보호자가 빠르게 해석할 수 있는 단순 문장형 요약
- 부부 공동 사용 고려
  - 동일한 기록 상태를 함께 확인 가능하도록 설계
- 모바일 앱 유사 사용성
  - 홈 화면 추가
  - 전체 화면에 가까운 UI
  - 큰 버튼 중심의 빠른 입력 UX

## 저장 방식 전략

### 1차 전략: 내부 저장소 중심

초기 MVP에서는 서비스 내부 저장소에 데이터를 저장하고 처리합니다.

- 장점
  - 구현 속도가 빠르다.
  - 인증, 권한, 외부 연동 복잡도를 줄일 수 있다.
  - 분석 로직과 기본 UX 검증에 집중할 수 있다.
- 고려사항
  - 사용자 2인이 함께 쓰는 경우 동기화 구조가 필요하다.
  - 기기 변경, 데이터 백업, 복구 요구에 한계가 있다.
  - 알림 기능 구현 시 서버 또는 백그라운드 처리 제약을 함께 고려해야 한다.

### 향후 전략: 외부 저장소 또는 동기화 가능한 백엔드

서비스 사용성이 검증되면 외부 저장소 기반으로 확장합니다.

- 후보 방향
  - Firebase
  - Supabase
  - 계정 기반 데이터 동기화
  - 공유 가족 계정 또는 초대 기반 공동 기록
- 기대효과
  - 다중 기기 동기화
  - 안정적인 백업 및 복구
  - 알림 및 자동화 기능 확장
  - 향후 분석 고도화에 유리

## 알림 전략

마지막 수유 시간 기준으로 4시간 동안 업데이트가 없을 경우 알림을 목표로 합니다.

### MVP 관점

웹 기반 서비스에서는 iPhone 환경상 알림 구현에 현실적 제약이 있습니다.

- iOS Safari 및 PWA의 푸시 알림은 환경과 설치 조건의 영향을 받는다.
- 사용자가 홈 화면에 추가하고 알림 권한을 허용해야 한다.
- 백그라운드에서 정교한 타이머처럼 항상 동작하는 방식은 어렵다.
- 마지막 수유 후 4시간 규칙을 안정적으로 처리하려면 서버 측 스케줄링 또는 푸시 인프라가 사실상 필요할 수 있다.

### 현실적인 권장 방향

- 1차
  - 앱 또는 PWA 설치 유도
  - 알림 기능은 지원 가능 여부 검토 및 제한적 도입 수준으로 포함
  - 우선은 화면 내 경고 상태, 배지, 강조 UI로 대체 가능
- 2차
  - 푸시 알림 인프라 도입
  - 마지막 수유 시각 기준 서버 예약 알림 처리
  - 사용자별 알림 허용 여부 설정 제공

## 추천 기술 스택

MVP 1차 구현 스택은 `React + Vite + Supabase + Vercel`로 정합니다.

- 프론트엔드
  - React 기반 SPA
  - Vite를 사용한 빠른 개발 환경 구성
  - 모바일 우선 반응형 UI
- 스타일
  - Tailwind CSS 또는 경량 CSS 설계
- 저장소
  - 1차: Supabase
  - 향후: Supabase 기반 확장 또는 별도 백엔드 분리
- 배포
  - Vercel
- 앱 확장 전략
  - 우선은 웹사이트로 운영
  - 향후 동일한 웹 자산을 기반으로 iOS와 Android 앱 래핑 검토
- 알림
  - 초기: UI 경고 중심
  - 향후: 앱 전환 또는 서버 기반 알림 시스템 검토

## 기술 선택 이유

- React + Vite는 단일 페이지 MVP를 빠르게 구현하고 유지보수하기에 적합하다.
- Supabase는 무료 플랜에서 인증, 데이터베이스, 향후 동기화 확장을 함께 다루기 좋다.
- Vercel은 프론트엔드 배포와 미리보기 배포, 커스텀 도메인 연결이 단순하다.
- 이 조합은 웹 MVP를 먼저 출시한 뒤, 추후 iOS와 Android 앱 형태로 확장하기에도 유리하다.

## MVP 범위

다음 범위를 MVP로 정의합니다.

- 아이폰에서 사용 가능한 단일 페이지 UI
- 수유 시간 기록
- 대변 시간 기록
- 최근 기록 목록 및 마지막 기록 시점 표시
- 오늘 기준 간단한 요약 정보 제공
- Supabase 기반 데이터 저장 및 조회
- 모바일 브라우저 사용성을 고려한 웹앱 형태의 기본 구성

다음 항목은 MVP에서 제외하거나 제한적으로 다룹니다.

- 정교한 사용자 권한 체계
- 고급 통계 대시보드
- 안정적인 푸시 알림 자동화
- 네이티브 앱 수준의 백그라운드 동작

## 추후 확장

- 소변, 수면, 체온, 약 복용 등 기록 항목 확장
- 보호자별 입력 주체 구분
- 일별 및 주별 통계 리포트
- 차트 기반 패턴 분석
- 데이터 내보내기
- 가족 공유를 고려한 계정 기반 동기화
- 알림 세분화
  - 수유 주기 알림
  - 배변 이상 패턴 감지
  - 기록 누락 리마인드
- 네이티브 앱 전환 또는 하이브리드 앱 패키징 검토

## 리스크 및 제약

- iPhone 웹 환경에서는 네이티브 앱 수준의 알림, 백그라운드 동작, 시스템 통합에 제약이 있다.
- 부부 공동 사용을 안정적으로 지원하려면 결국 동기화 가능한 백엔드가 필요할 가능성이 높다.
- 초기 내부 저장소 구조가 향후 외부 저장소 이전을 어렵게 만들지 않도록 데이터 모델을 단순하고 명확하게 설계해야 한다.
- 입력 속도를 높이려다 보면 기록 항목의 정확성이나 확장성이 떨어질 수 있다.
- 모바일 웹은 빠르게 배포할 수 있지만, 실제 App Store 네이티브 앱과 동일한 사용성을 완전히 보장하지는 않는다.
- 알림 기능은 사용자 기대치가 높기 때문에 지원 범위와 동작 조건을 제품 설명에서 명확히 안내해야 한다.
